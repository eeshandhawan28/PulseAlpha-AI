Phase 3: LangGraph Topology

How should the analysis graph be structured? This affects latency, complexity, and how easy it is to debug failed runs.

A

Sequential Pipeline

Nodes run one after another in a fixed order. Simple to reason about, easy to resume from any checkpoint. Each node reads from and writes to AnalysisState.

ingest_fundamentals → ingest_market_data → ingest_fii_dii → ingest_sentiment → ingest_ipo_gmp
→ compute_rrg → compute_flow_strength → compute_divergence
→ normalize_and_validate → END

Pros

  • Dead-simple resume/retry
  • Easy to test each node in isolation
  • Deterministic execution order

Cons

  • Slower — connectors run serially
  • One slow connector blocks everything
B

Fan-out / Fan-in DAG

Ingestion nodes run in parallel via LangGraph's Send() API, then a fan-in node aggregates results before feature computation.

START → [ingest_fundamentals ‖ ingest_market_data ‖ ingest_fii_dii ‖ ingest_sentiment ‖ ingest_ipo_gmp]
→ aggregate_ingest
→ [compute_rrg ‖ compute_flow_strength] → compute_divergence
→ normalize_and_validate → END

Pros

  • Faster — connectors run concurrently
  • Better use of async I/O

Cons

  • LangGraph Send() adds complexity
  • Harder to debug partial failures
  • Resume logic is more involved
C

Sequential + Async Connectors

Graph nodes run sequentially (simple topology), but each ingestion node calls its connectors with asyncio.gather() internally — parallelism inside the node, not in the graph.

ingest_all_data [internally: gather(fundamentals, market, fii_dii, sentiment, gmp)]
→ compute_features [internally: gather(rrg, flow_strength)]
→ compute_divergence → normalize_and_validate → END

Pros

  • Fast (async gather) + simple graph topology
  • Easy resume — just 4 nodes to checkpoint
  • Best of both worlds

Cons

  • Less granular checkpointing per connector
  • One bad connector can fail the whole ingest node
